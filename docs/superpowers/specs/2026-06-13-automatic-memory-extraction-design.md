# AstrBot 对话便签自动提取设计

## 目标

在现有私聊长期便签插件中增加自动提取能力。插件先缓冲一段完整对话，在消息数量或空闲时间达到可配置阈值后，调用模型统一整理长期有用的信息，并静默保存为可见、可编辑、可删除的便签。

## 范围

第一版自动提取仅处理私聊，包含：

- 缓冲用户消息和机器人最终回复；
- 按消息数量或空闲时间触发整理；
- 使用当前会话模型或可选的专用模型；
- 从对话中提取少量长期有用的便签；
- 自动分类、校验、去重和更新；
- 整理失败后保留缓冲并重试；
- 在现有 WebUI 中标识自动生成便签和整理状态。

不包含群聊记忆、向量数据库、跨用户合并、聊天内确认指令或自动删除人工便签。

## 用户体验

- 自动提取默认开启，静默运行，不向用户发送“已记住”之类的提示。
- 默认在以下任一条件满足时整理：
  - 当前用户累计 20 条未处理消息；
  - 最后一条未处理消息后空闲 30 分钟。
- 用户和机器人消息都计入消息数。
- 只有存在未处理消息时才检查和调用模型。没有新消息时绝不触发整理请求。
- 管理员仍在现有“对话便签”页面查看、搜索、编辑和删除全部便签。
- 自动生成的便签显示来源标识，但编辑和删除方式与人工便签一致。

## 配置

新增配置项：

- `auto_extract_enabled`：启用自动提取，默认 `true`。
- `auto_extract_idle_minutes`：空闲触发分钟数，默认 `30`。
- `auto_extract_message_threshold`：消息数触发阈值，默认 `20`。
- `auto_extract_provider_id`：可选专用模型提供商 ID。为空时使用产生该会话回复的当前模型。
- `auto_extract_max_notes`：单次最多新增或更新的便签数，默认 `5`。
- `auto_extract_retry_minutes`：失败后的重试间隔，默认 `10`。

所有数值在运行时设置合理边界，防止零间隔循环或单次生成过多便签。

## 数据模型

### 对话缓冲

新增 `message_buffer` 表：

- `id`：自增主键；
- `scope_key`：平台实例 ID 与私聊用户 ID 组成的隔离键；
- `role`：`user` 或 `assistant`；
- `content`：纯文本消息；
- `provider_id`：该轮会话实际使用的模型提供商 ID，可为空；
- `created_at`：消息时间；
- `batch_id`：为空表示待整理；非空表示正在整理或已归属某次批次。

只保存自动提取所需的短期缓冲，不把它混入 AstrBot 的长期聊天记录。成功整理后删除对应批次消息；失败时释放批次，使消息可以重试。

### 整理状态

新增 `extraction_state` 表：

- `scope_key`：用户隔离键；
- `last_message_at`：最后一条待处理消息时间；
- `next_retry_at`：失败后的最早重试时间；
- `processing_batch_id`：当前正在处理的批次；
- `last_error`：最近一次错误摘要；
- `last_extracted_at`：最近成功整理时间。

数据库状态用于保证 AstrBot 重启后仍可继续整理，并防止同一用户并发生成重复任务。

### 便签来源

扩展 `notes` 表：

- `source`：`manual` 或 `auto`，旧数据迁移为 `manual`；
- `source_batch_id`：自动整理批次 ID，可为空。

数据库 schema 版本升级，并提供从现有版本的原地迁移。

## 事件与数据流

### 用户消息

现有私聊监听器登记用户后，将有效纯文本消息写入缓冲。空消息、机器人内部事件、群聊和缺失身份的事件不写入。

### 机器人回复

使用 AstrBot 4.25.5 的 LLM/Agent 完成后钩子获取最终回复文本和实际提供商。只有正常私聊回复写入缓冲，工具中间结果和空回复不写入。

### 触发方式

插件维护一个轻量后台调度任务，只查询数据库中存在待处理消息的用户：

1. 如果待处理消息数达到配置阈值，立即安排整理。
2. 否则，如果最后消息时间距当前时间达到空闲阈值，安排整理。
3. 如果用户没有待处理消息，不产生模型请求。
4. 如果仍在失败退避期，不安排整理。
5. 同一 `scope_key` 同时最多运行一个整理任务。

调度器只负责检查数据库和启动任务，不阻塞正常聊天回复。

## 模型调用

整理器优先使用 `auto_extract_provider_id` 指定的模型。未指定时，使用缓冲中最近一次机器人回复记录的提供商；仍无法确定时跳过本轮并保留缓冲，不猜测其他模型。

模型收到：

- 按时间排序的用户和机器人对话；
- 当前用户已有便签的精简列表；
- 固定的提取规则；
- 严格 JSON 输出协议。

模型只能返回候选数据，不能直接操作数据库。期望结构：

```json
{
  "memories": [
    {
      "action": "create",
      "category": "偏好",
      "content": "用户喜欢简洁直接的回答"
    },
    {
      "action": "update",
      "note_id": 12,
      "category": "目标",
      "content": "用户正在准备六月的英语考试"
    }
  ]
}
```

允许的操作只有：

- `create`：创建新自动便签；
- `update`：更新属于当前用户的已有便签；
- 不返回任何内容：本段对话没有值得长期保存的信息。

模型不得删除便签。所有字段由插件再次校验，超过单次上限的候选被截断。

## 提取准则

值得保存：

- 稳定偏好和明确雷区；
- 反复出现的习惯；
- 重要人物及关系；
- 有持续意义的事件、考试和日程；
- 当前项目、阶段目标和待办；
- 用户明确要求长期记住的信息。

不保存：

- 普通问答和临时闲聊；
- 模型自己的推测；
- 仅由机器人提出、用户未确认的事实；
- 密码、验证码、令牌、Cookie 和其他凭据；
- 过度敏感且与长期陪伴无必要关系的信息；
- 已有便签的同义重复。

## 去重与更新

插件在写入前执行两层处理：

1. 规范化内容后完全一致：跳过创建。
2. 模型明确返回属于当前用户的 `note_id`：更新该便签。

模型会看到已有便签，因此应优先更新已经变化的事实。插件不依赖模糊字符串算法自动覆盖人工便签，避免错误合并；无法可靠判断时宁可创建一条可见便签，由管理员在后台处理。

人工便签和自动便签都可被模型建议更新，但更新必须携带当前用户范围内的有效 ID。模型不能跨用户引用便签。

## 并发与失败处理

- 领取批次和标记 `processing_batch_id` 在同一数据库锁内完成。
- 同一用户的新消息可以在旧批次整理时继续写入，归入下一批。
- 成功保存候选后，在同一业务流程中删除已处理批次并更新状态。
- 模型超时、JSON 无效、提供商不可用或数据库失败时，不影响当前聊天。
- 失败批次释放回待处理状态，并设置 `next_retry_at`，到退避时间后重新检查。
- 插件卸载或 AstrBot 停止时取消调度器；已经启动的模型任务被取消后释放批次。
- 日志只记录错误摘要和批次信息，不记录完整私聊正文。

## WebUI

沿用现有卡片布局，增加：

- 自动便签的小型“自动生成”标识；
- 用户区域中的待整理消息数、最后消息时间和最近整理状态；
- 配置仍由 AstrBot 插件配置页管理，不在自定义页面重复实现。

页面 API 继续复用 AstrBot Dashboard 鉴权。响应避免使用会被 Plugin Page Bridge 解包的顶层 `data` 字段，预期错误继续使用 Bridge 可读的错误信封。

## 安全与隐私

- 只处理私聊，并继续使用平台实例 ID + 用户 ID 隔离。
- 缓冲内容只保存在插件数据目录的 SQLite 中。
- 不记录凭据类信息为便签；提取提示词明确禁止保存凭据。
- 模型输出作为不可信数据处理，执行类别白名单、长度限制、用户范围校验和操作白名单。
- 自动便签注入模型时仍经过现有 XML 边界转义，并作为临时内容块使用。

## 测试

测试覆盖：

- 用户和机器人消息按用户隔离写入缓冲；
- 没有待处理消息时不触发；
- 消息数量阈值触发；
- 空闲时间阈值触发；
- 两个条件均未满足时不触发；
- 指定模型和当前会话模型的选择顺序；
- 严格 JSON 解析和无效输出降级；
- 凭据类候选拒绝；
- 完全重复候选跳过；
- 带有效 ID 的同用户便签更新；
- 跨用户 ID 更新拒绝；
- 成功后清理批次；
- 失败后保留并退避；
- 重启后继续处理；
- 群聊、空消息和缺失身份不缓冲；
- 自动便签 WebUI 标识；
- 原有手动管理和相关便签注入行为不回归。
