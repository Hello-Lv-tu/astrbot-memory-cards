# 对话便签提取质量与去重合并设计

日期：2026-06-19

## 目标

提升自动便签的分类、去重和冲突更新质量。语义相近的信息默认收敛为一条更完整的有效便签；新旧信息冲突时更新原便签，并保留可追溯的变更历史。

本次还为现有便签提供一次可预览、需确认的质量整理，但不引入向量数据库、一键回滚或自动删除历史。

## 方案

采用“本地候选召回 + 模型判定 + 程序校验”的混合流程。

只优化提示词无法稳定处理近义重复；引入向量模型则会增加模型费用、配置和运维负担。混合流程先用分类、关键词和字符片段从当前用户的有效便签中召回少量候选，再允许模型在候选范围内输出结构化操作，因此能兼顾语义质量、确定性和部署成本。

## 提取流程

1. 从一批私聊消息中提取以用户为主体、明确表达且长期有用的信息。
2. 为每条待判断信息召回同一用户的候选旧便签。
3. 模型只能输出 `create`、`update`、`merge` 或 `noop`：
   - `create`：不存在相关旧便签时新增。
   - `update`：新信息补充或推翻单条旧便签时更新原记录。
   - `merge`：多条近义便签收敛为一条完整表述。
   - `noop`：信息短期、含糊、重复但无需改写，或无法安全判断。
4. 程序验证分类、内容、候选 ID、用户归属和敏感信息，再在事务中应用操作。
5. 操作成功后完成消息批次；解析、模型或事务失败时不改便签，并沿用现有重试机制。

模型不能引用未提供的便签 ID，也不能操作其他用户的数据。模型自述、助手推测、一次性闲聊和低置信信息不落库。

## 合并与冲突规则

- 语义相近的新信息默认更新原便签，不创建并列重复项。
- 多条旧便签合并时保留最早创建的便签 ID，写入更完整且不过度推断的内容。
- 新信息明确否定旧信息时更新旧便签，不同时保留互相冲突的有效内容。
- 被替换或合并移出的内容先写入变更历史；历史不参与检索和注入。
- 所有写操作必须绑定当前 `scope_key`，并在单个数据库事务内完成。

## 数据模型

数据库从 v2 升级到 v3，新增 `note_revisions` 表，至少保存：

- 当前用户范围 `scope_key`
- 保留便签 ID 与被合并便签 ID（适用时）
- 变更前分类和正文
- 变更类型与原因
- 来源批次 ID
- 变更时间

现有 `notes` 继续只保存当前有效便签。升级不得改写已有 v2 便签；整理只在管理员确认预览后发生。

## 质量整理预览

后台“质量整理”入口按用户生成预览，显示重分类、改写、合并和忽略项。预览包含基于当前便签集合生成的版本指纹；应用时重新校验指纹，若数据已变化则拒绝执行并要求重新生成。

预览生成失败或过期不会修改任何便签。确认应用后，所有操作在一个事务中完成。第一版支持查看变更历史，不提供一键回滚。

## 界面与接口

在现有插件页面加入：

- “质量整理”按钮与预览面板
- 每项操作的类型、旧内容和建议内容
- 确认应用与取消入口
- 只读的便签变更历史视图

新增接口继续使用插件命名空间并复用 Dashboard 鉴权。接口负责生成预览、应用预览和读取历史，不接受前端提交的任意数据库操作列表作为可信输入。

## 错误与安全

- 任何提取错误都不得阻塞正常聊天回复。
- 非法 JSON、越权 ID、未知分类、空内容、敏感凭据和超长内容均拒绝落库。
- 合并失败必须整体回滚，不允许留下半合并状态。
- 现有私聊隔离规则保持不变，群聊不读取、不整理、不注入。
- 线上现有数据只用于人工分析，不发送给委派代理或写入测试夹具。

## 验证

测试优先覆盖：

- 近义偏好合并为一条有效便签
- 新旧信息冲突时更新并记录历史
- 不相关信息保持独立
- 跨用户或非候选 ID 操作被拒绝
- 非法模型输出不落库
- 合并事务失败时完整回滚
- 预览过期保护
- v2 数据无损升级到 v3
- 现有自动整理、检索、Web API 和插件生命周期测试无回归

完成前还需运行完整测试、静态检查、编译检查，并在实际 AstrBot 环境检查加载、重载、数据库持久化与页面/API 鉴权。
